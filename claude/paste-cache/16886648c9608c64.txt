今天这篇文章源于最近半年深度使用 Claude Code、两个账号每月 40 刀氪金换来的一些踩坑经验。研究了 Claude Code 本身之后才意识到，这不是 Prompt 问题，而是这套系统的设计就是这样的。

层职责CLAUDE.md / rules / memory长期上下文，告诉 Claude "是什么"Tools / MCP动作能力，告诉 Claude "能做什么"Skills按需加载的方法论，告诉 Claude "怎么做"Hooks强制执行某些行为，不依赖 Claude 自己判断Subagents隔离上下文的工作者，负责受控自治Verifiers验证闭环，让输出可验、可回滚、可审计

图表六：Plan Mode 图（Section 3）
阶段特征探索阶段 (PlanMode)只读操作为主，澄清目标，提交方案，价值：将探索与执行在"副作用"层面隔离，降低错误方向的自动成本执行阶段产生实际变更，消耗 Token，产生历史记录
配合命令：/clear（切断失败路径）、/compact（压缩历史）、--fork-session/--worktree（并行隔离）

图表十二：Section 13 常见反模式表
反模式症状修复CLAUDE.md 当 wiki每次加载污染上下文，关键指令令被稀释只保留契约的，资料拆到 Skills 和 rulesSkill 大杂烩描述过于模糊，工作流冲突一个 Skill 只做一件事，副作用显式控制工具太多描述模糊选错工具，schema 污染上下文合并同类工具，细明 namespacing没有验证闭环Claude 只能"觉得自己完成了"给每类任务明定 verifier过度自治多 agent 并行无边界，出错难定位角色/权限/worktree 最小化，限制 maxTurns上下文不做切分研究、实现、审查全堆在主线程，有效上下文被稀释任务切换 /clear，阶段切换 /compact，重型探索交给 subagent（Explore → Main 模式）自动范围过宽治理不足多 agent，外部工具全开，但缺乏权限边界和结果回收逻辑permissions + sandbox + hooks + subagent 组合边界已批准命令令触发不清晰settings.json 里残留 rm -rf 等危险操作，一旦触发不可逆定期审查 .claude/settings.json 的 allowedTools 列表
